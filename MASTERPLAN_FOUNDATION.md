# ChessCoin — Мастер-план основы (масштаб, безопасность, продакшн)

> Версия документа: **1.1** (март 2026)  
> Назначение: единая дорожная карта **фундамента** — чтобы backend, БД и frontend согласованно росли к **высокой нагрузке** без обрушения системы при **постепенном** росте аудитории (в т.ч. к порядка **1M активных пользователей** в перспективе).  
> Связь: дополняет `MASTERPLAN.md` (фичи и версии); этот файл — **стратегия платформы и инженерные принципы**.

---

## 1. Стратегическое решение: не «с нуля», а **эволюция с фундаментом**

### 1.1 Сравнение вариантов

| Критерий | Переписать с нуля | Развивать текущий код |
|----------|-------------------|------------------------|
| Срок до «как сейчас по фичам» | Много месяцев, высокий риск регрессий логики | Короче: логика игры, биржа, войны уже в коде |
| Риск потери деталей | Высокий (краевые случаи, экономика, TON) | Ниже при поэтапном рефакторинге |
| Масштабирование | Не гарантируется «с нуля» без той же архитектуры | Достигается **паттернами** (stateless, пулы, реплики, очереди) |
| Команда / один разработчик | Переписывание часто не окупается | **Итерации + метрики** дают результат быстрее |

### 1.2 Рекомендация

**Не переписывать проект целиком.** Текущий стек (Express, Socket.io, Prisma, PostgreSQL, Redis, Vite/React) **подходит** для роста, если правильно выстроить **границы ответственности**, **наблюдаемость** и **масштабирование по горизонтали**.

**Имеет смысл «как с нуля» переписать только узкие модули** после стабилизации, если доказано узкое место (например, отдельный **сервис матчмейкинга** или **вынесенный воркер Stockfish**), а не весь монолит.

**Параллельная линия:** исправить **критические ошибки** (сборка frontend, безопасность секретов, согласованность `updateBalance`, middleware админки) — это **не опционально** ни для «мало игроков», ни для масштаба.

## 2. Что значит «1 миллион игроков» в инженерных терминах

Один миллион **зарегистрированных** и один миллион **одновременно активных** — разные задачи. План ориентируем на **рост нагрузки во времени**:

- **DAU / пиковые одновременные сессии** и **соединения WebSocket** задают нагрузку на **CPU, память, файлы дескрипторов, Redis, Postgres**.
- «Не рухнет» означает: **лимиты известны**, **автомасштабирование или добавление инстансов** по метрикам, **деградация контролируемая** (очереди, rate limit), а не молчаливый отказ.

**Вывод:** цель — не «один сервер на 1M», а **архитектура, которую можно наращивать**: несколько инстансов API, репликация БД, кластер Redis при необходимости, CDN для статики, фоновые очереди для тяжёлых операций.

## 3. Принципы целевой архитектуры (фундамент)

### 3.1 Backend

- **Stateless HTTP**: любой инстанс может обслужить запрос; сессия — JWT / внешнее хранилище, не память процесса.
- **Socket.io**: уже с **Redis adapter** — сохранить; при росте — **sticky sessions** на балансировщике или единый кластер Redis с нормальной **политикой памяти и персистентности** (по необходимости).
- **Единая финансовая модель**: все изменения баланса и связанных агрегатов — через **один слой** (например, `updateBalance` + транзакции БД), без «тихих» прямых `balance +=` в разных файлах.
- **Идемпотентность** платёжных и критичных операций (TON, исполнение ордеров) — ключи/статусы в БД.
- **Очереди** (BullMQ / отдельный worker) для: тяжёлых рассылок, массовых начислений, длительной обработки — чтобы не блокировать API.

### 3.2 База данных (PostgreSQL)

- **Connection pooling** (PgBouncer или пул на стороне приложения с лимитами) — обязателен до роста конкуренции соединений.
- **Индексы** под реальные запросы (лидерборды, сессии, ордера); периодический **EXPLAIN** на горячих путях.
- **Read replicas** для тяжёлого чтения (рейтинги, аналитика) — на этапе заметной нагрузки.
- **Миграции** только через Prisma migrate; **бэкапы** и **план восстановления** — часть продакшн-чеклиста.

### 3.3 Redis

- Чёткое разделение: **кеш**, **pub/sub Socket.io**, **rate limiting**, **краткоживущие блокировки** — при росте возможен **отдельный инстанс/DB index** под разные роли.
- Политика **eviction** и **maxmemory** согласована с допустимой потерей кеша vs стабильностью сокетов.

### 3.4 Frontend

- Сборка **без ошибок TypeScript** в CI — обязательный шлюз перед деплоем.
- Минимизация тяжёлых ререндеров в игре; **разделение** тяжёлых экранов (lazy routes).
- Статика и ассеты — **nginx/CDN**, долгий кеш для `assets`.

### 3.5 Безопасность

- Секреты только в **env / Secrets Manager**, не в git.
- **Ротация** скомпрометированных токенов (бот, webhook).
- Валидация **Telegram initData** с осмысленным **окном времени** (`expiresIn`), без debug-обхода в продакшене.
- **Rate limiting** на auth и дорогих эндпойнтах; **helmet/CORS** согласованы с доменом Mini App.
- **Единая модель прав** для админ-маршрутов (проверка роли из БД, а не флаги только в `req.user` без загрузки).

### 3.6 Наблюдаемость (без этого масштаб слепой)

- Структурированные **логи**, **корреляционный id** для запросов.
- **Метрики**: latency p95/p99, ошибки 5xx, загрузка CPU/RAM, **подключения к PG/Redis**, длина очередей.
- **Алерты** на health-check, диск, рестарты контейнеров.

### 3.7 Критерии выноса Stockfish и матчмейкинга (точечный рефакторинг)

Отдельные модули **не выносятся «на будущее»** — только когда **метрики** показывают узкое место. До этого оставляем пул воркеров и логику батлов в текущем backend.

**Stockfish (движок ИИ)**

- **Оставляем в монолите**, пока: CPU основного процесса не упирается в расчёт ходов, очередь к воркерам стабильна, задержка хода ИИ в пределах целевого UX.
- **Имеет смысл вынос** (отдельный сервис или контейнер только с воркерами, запрос по очереди/Redis/RPC), если:
  - рост **задержки** хода ИИ при росте одновременных партий;
  - **загрузка CPU** API-процесса в основном из Stockfish;
  - нужно **масштабировать только шахматный CPU** без дублирования всего backend.

**Матчмейкинг (подбор соперника)**

- **Оставляем в монолите** (games + Redis), пока: подбор по комнатам/коду/простым правилам не даёт заметной задержки остальным запросам.
- **Имеет смысл вынос** (отдельный модуль или сервис с собственными лимитами и метриками), если:
  - появляются **рейтинговые очереди**, сложные правила подбора, высокая частота операций поиска;
  - профилирование показывает, что **подбор** занимает существенную долю времени обработки или блокирует event-loop/API.

**Общее правило:** сначала **наблюдаемость** (метрики по latency, CPU, длине очередей), потом решение о разделении — иначе растёт сложность деплоя без выигрыша.

## 4. Дорожная карта по фазам

### Фаза 0 — Стабильность и продакшн-готовность «для малых нагрузок» (обязательный фундамент)

**Цель:** приложение **целиком функционально**, сборка честная, секреты под контролем.

- Устранить **критические** ошибки frontend (TS, runtime в игре/войнах/профиле).
- Выровнять **админку и airdrop**, **initData**, убрать debug из прода.
- CI: **lint + `tsc` + тесты** перед деплоем (хотя бы backend-тесты + frontend build).
- Документировать **переменные окружения** и **процедуру деплоя**.

**Критерий выхода:** зелёный pipeline, нет известных блокирующих багов в основных сценариях.

---

### Фаза 1 — Безопасность и целостность данных

**Цель:** доверие к деньгам и аккаунтам.

- Свести денежные операции к **одному слою** + аудит транзакций.
- Проверить **идемпотентность** TON/биржи.
- Ревизия **прав доступа** (админ, бот-секреты, webhook).

**Критерий выхода:** нет рассинхрона «баланс vs история» в типовых сценариях; секреты вне репозитория.

---

### Фаза 2 — Масштабируемость «первого порядка» (тысячи–десятки тысяч одновременных)

**Цель:** выдержать рост без смены стека.

- Ввести **PgBouncer** (или эквивалент) и лимиты пула в приложении.
- Нагрузочное тестирование **API + WebSocket** (хотя бы synthetic).
- Вынос тяжёлой работы в **очередь**; при необходимости — отдельный **worker**-контейнер.
- Проверка **горизонтального** масштаба: 2+ инстанса backend за балансировщиком (даже если временно на одном VPS — проверить конфиг).

**Критерий выхода:** документированные **SLO** (например, p95 API < X ms при N RPS) и отсутствие очевидных узких мест в одном потоке.

---

### Фаза 3 — Рост к очень большой аудитории (порядок 1M+ пользователей в базе / высокий DAU)

**Цель:** не «одна машина», а **проверенная** топология.

- **Read replicas** PostgreSQL, оптимизация самых дорогих запросов.
- **Redis**: при необходимости Sentinel/кластер; разделение нагрузки.
- **Шардирование** или выделенные сервисы только там, где **метрики** показали потолок (часто — отдельно матчмейкинг, отдельно биржа).
- **CDN** и агрессивный кеш для статики.

**Критерий выхода:** устойчивость к отказу одного узла (частично), план **capacity** на квартал.

## 5. Синхронизация backend ↔ БД ↔ frontend

| Слой | Обязательства |
|------|----------------|
| **API** | Версионирование или согласованные DTO; не ломать контракт без миграции и фронта. |
| **Prisma schema** | Единственный источник правды по моделям; миграции ревьюятся. |
| **Frontend** | Типы из `api.ts` / общий пакет типов при необходимости; без «магических» строк для статусов игры. |
| **Сокеты** | Документированные события и payload; согласование с `GameSession` на клиенте. |

Периодически: **сквозной сценарий** (логин → игра → завершение → баланс) в staging.

## 6. Итоговая рекомендация «что делать сейчас»

1. **Зафиксировать** этот документ как **обязательный ориентир** для всех крупных изменений.  
2. **Не** начинать полный rewrite — **сначала Фаза 0–1** на текущем репозитории.  
3. **Параллельно** вести `MASTERPLAN.md` по фичам; крупные рефакторинги заводить как **задачи с критериями** из разделов 3–4.  
4. Пересмотреть **полный rewrite** только если после Фазы 2 метрики и команда покажут, что **монолит** мешает сильнее, чем выделение сервисов по границам.
