# ChessCoin — План реконструкции системы (без переписывания с нуля)

> Версия: **1.0** (март 2026)  
> Контекст: после «оптимизационных» правок и деплоя с нуля на сервере участились регрессии; рабочая логика игры частично сохранилась в коде, но **стабильность и целостность** нужно восстановить поэтапно.  
> Назначение: **единый нумерованный порядок работ**, разделение **важности** и **срочности (последовательности)**, **критерии проверки** перед релизом и перед переходом к следующему шагу.

Связанные документы: [MASTERPLAN.md](./MASTERPLAN.md), [MASTERPLAN_FOUNDATION.md](./MASTERPLAN_FOUNDATION.md), [AUDIT.md](./AUDIT.md).

---

## Как читать этот план

### Важность (фундамент vs остальное)

| Уровень | Обозначение | Смысл |
|---------|-------------|--------|
| **Фундамент** | `[Ф]` | Без этого **нельзя** считать систему стабильной и готовой к масштабу: риск утечек, порчи данных, неработающего ядра игры. |
| **Высокая** | `[В]` | Сильно влияет на качество, безопасность и сопровождение; желательно до «широкого» продакшна. |
| **Средняя** | `[С]` | UX, i18n, чистота кода; можно параллелить после фундамента. |

### Срочность (порядок — «без чего нельзя к следующему»)

**Срочность** здесь = **номер шага U1 → U2 → …** Выполнять в этом порядке, чтобы не переделывать работу (например, не чинить UI до вывода секретов из git и не масштабировать до единой экономики).

## Нумерованная последовательность задач

### U1. Секреты и следы доступа вне репозитория `[Ф]` · срочность: **1**

**Зачем первым:** пока токены и SSH-данные в истории git, любой с доступом к репо — риск; дальнейшие правки и пуши **опасны**; ротация после правок всё равно нужна.

**Содержание:**
- Удалить из трекинга чувствительные файлы (напр. `.claude/settings.local.json` с токеном бота, IP, SSH).
- Добавить `.claude/` (и аналоги) в `.gitignore`.
- **Ротировать** токен бота в @BotFather; обновить секреты на сервере и в CI.
- При необходимости — очистка истории (`git filter-repo` / BFG) *или* принятие риска + ротация всех засвеченных секретов.

**Критерии проверки (Done):**
- [ ] В рабочем дереве и в **последнем коммите** нет полного `BOT_TOKEN` и паролей.
- [ ] `.gitignore` закрывает локальные настройки с секретами.
- [ ] Бот на проде работает с **новым** токеном.
- [ ] Запись в AUDIT: C5 закрыт или перенесён в «история очищена / секреты сменены».

**Связь:** AUDIT C5, H5.

### U2. Шлюз качества в CI: сборка и тесты перед деплоем `[Ф]` · срочность: **2**

**Зачем вторым:** иначе снова можно задеплоить «сломанный» фронт или backend; повторится сценарий «с нуля на сервере + глюки».

**Содержание (реализовано в `.github/workflows/deploy.yml`):**
- **verify-backend:** `npm ci` → `prisma generate` → `npm test` → `tsc --noEmit`.
- **verify-frontend:** `npm ci` → `npm run build` (Vite). Полный `tsc` на фронте в CI — **после U8**, пока в проекте много ошибок TS; локально: `npm run typecheck` в `frontend/`.
- **deploy** запускается только после успеха обеих verify; для **pull request** в `main` выполняются только verify, без деплоя.

**Критерии проверки:**
- [ ] Push в `main`: при падении тестов или сборки фронта деплой **не** выполняется.
- [ ] Локально воспроизводимы команды из README (раздел «CI и проверка перед релизом»).
- [ ] После U8: добавить в CI шаг `npm run typecheck` во frontend (опционально отметить здесь).

**Связь:** AUDIT M15; MASTERPLAN_FOUNDATION фаза 0.

### U3. Критические runtime-ошибки ядра игры (frontend) `[Ф]` · срочность: **3**

**Зачем третьим:** без работающей доски и попапов войны «игра» не воспроизводима; дальнейшая оптимизация бессмысленна.

**Содержание:**
- `WarChallengePopup`: вызов `const t = useT()`, исправить строку подзаголовка (i18n), комментарий про 30 сек.
- `ChessBoard`: объявить `useState` для `pendingPromotion` / `setPendingPromotion`, согласовать с `PromotionModal`.
- Устранить **конфликт дублирующих экспортов** в `frontend/src/types/index.ts` (типы из `./api` vs локальные интерфейсы).

**Критерии проверки:**
- [ ] `npx tsc --noEmit` в `frontend` **без ошибок** в перечисленных файлах (цель — весь фронт без ошибок на шаге U8).
- [ ] Ручной прогон: партия с **превращением пешки** не падает.
- [ ] Ручной прогон: попап вызова на войну открывается без краша.

**Связь:** AUDIT C1, C2; частично типы в H6.

### U4. Безопасность входа и админ-маршрутов (backend) `[Ф]` · срочность: **4**

**Зачем четвёртым:** до исправления нельзя полагаться на airdrop и прод-настройки auth.

**Содержание:**
- `airdrop.ts`: привести `adminOnly` к той же модели, что в `admin.ts` — проверка `isAdmin` **из БД** по `req.userId` / `req.user.id`, а не `req.user?.isAdmin`.
- `auth.ts`: убрать или жёстко ограничить **debug**-логин в продакшене; задать осмысленный `expiresIn` для `validate(initData)` (не бессрочный replay).
- Проверить дефолты `BOT_API_SECRET` и вебхуков — не «заглушки» в проде.

**Критерии проверки:**
- [ ] Запрос к `/admin/airdrop/*` с JWT обычного пользователя → **403**; с админом из БД → **200/нужный ответ**.
- [ ] В проде `DEBUG` не включает обход Telegram-подписи.
- [ ] Краткая запись в AUDIT: C6/H4/H7 закрыты или пересмотрены.

**Связь:** AUDIT C6, H4, M7.

### U5. Секция «Войны»: модалки и типы `[В]` · срочность: **5**

**Критерии проверки:**
- [ ] `tsc` без ошибок в `components/wars/*` и `api/index.ts`.
- [ ] Ручной прогон: открытие экрана войн, объявление/детали — без белого экрана.

**Связь:** AUDIT C3, C4, H6.

### U6. Единая экономика: свести баланс к `updateBalance()` `[Ф]` · срочность: **6**

**Зачем шестым:** пока деньги обходят центральный слой, масштабирование и аудит **ломаются**; правки в U7–U8 могут снова размножить расхождения — поэтому **до** массовой чистки TS и новых фич.

**Содержание:**
- Пройти маршруты из AUDIT H1: `exchange`, `gameTasks`, `tasks`, `tournaments`, `wars`, `cleanup`, `crons` — заменить прямые `prisma.user.update({ balance })` на `updateBalance` (или обёртку с транзакцией), где применимо.
- Зафиксировать исключения в коде-комментарии + тест, если исключение неизбежно.

**Критерии проверки:**
- [ ] Поиск по backend: нет «тихих» изменений `balance` вне `economy`/согласованных транзакций (кроме явно задокументированных).
- [ ] Тесты экономики проходят; выборочный сценарий: биржа / награда / отмена — баланс и `Transaction` согласованы.

**Связь:** AUDIT H1, MASTERPLAN_FOUNDATION §3.1.

### U7. Контракт API: `formatUser` и поля профиля `[В]` · срочность: **7**

**Зачем:** фронт ожидает `wins`/`losses`/ранг; иначе логика отображения и статистики **логически** неверна даже при зелёном билде.

**Содержание:**
- `backend/src/routes/auth.ts` (и связанные): `formatUser` возвращает стабильный контракт (wins, losses, rank по договорённости с типами frontend).
- Синхронизировать типы `UserPublic` / `api.ts` на фронте.

**Критерии проверки:**
- [ ] В ответе `/auth/me` (или эквиваленте) поля согласованы с типами фронта.
- [ ] Профиль и лидерборд не показывают `undefined` там, где ожидаются числа.

**Связь:** AUDIT H2.
