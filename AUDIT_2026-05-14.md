## КРИТИЧЕСКИЕ РЕГРЕССИИ (требуют немедленного хирургического фикса)

### A. Prisma enum `TransactionType` рассинхронизирован со схемой

В `backend/prisma/schema.prisma` объявлено **36 значений** `TransactionType`.
В реальной БД prod — **31 значение**. Отсутствуют пять:

| Отсутствует | Что ломает |
|---|---|
| `TOURNAMENT_WIN` | **cron `checkTournamentResults`** падает каждый час → турниры не выдают призы |
| `BATTLE_DONATION` | `battle:donate` не работает — донаты зрителей в публичном батле |
| `CLAN_WAR_WIN` | клановые войны не выдают трофеи |
| `TON_DEPOSIT` | пополнение через TON не работает |
| `WALLET_UNLOCK` | разовый платёж за подключение кошелька (1 TON) не списывается |

**Доказательство:** в логах backend каждые 60 мин:
```
prisma:error invalid input value for enum "TransactionType": "TOURNAMENT_WIN"
at Cron/Tournaments → checkTournamentResults
```

**Причина:** миграции `20260509_*` и `20260510_*` я применила вручную через
`ALTER TABLE ... ADD COLUMN`, но **не выполнила `prisma migrate deploy`** на
prod. История миграций в `_prisma_migrations` рассинхронизирована со схемой,
а enum-значения которые лежат в более ранних миграциях — пропущены.

**Фикс (когда Кенан скажет):**
```sql
ALTER TYPE "TransactionType" ADD VALUE 'TOURNAMENT_WIN';
ALTER TYPE "TransactionType" ADD VALUE 'BATTLE_DONATION';
ALTER TYPE "TransactionType" ADD VALUE 'CLAN_WAR_WIN';
ALTER TYPE "TransactionType" ADD VALUE 'TON_DEPOSIT';
ALTER TYPE "TransactionType" ADD VALUE 'WALLET_UNLOCK';
```
Безопасно. Один транзакционный запрос, никаких данных не теряет.
**Это не я сломала сегодня — это давняя инфра-регрессия (минимум с 25.04).**

### C. `checkDailyLoginTask` race-condition

```
PrismaClientKnownRequestError P2002 CompletedTask (userId, taskId)
```

Cron вызывается параллельно с user-action или дублируется → попытка вставить
уже существующую запись `CompletedTask`. Не критично, но засоряет логи.

**Фикс:** `upsert` вместо `create`.
