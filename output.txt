БЛОК 2: АТОМАРНОСТЬ И ЦЕЛОСТНОСТЬ ДАННЫХ (фундамент экономики)
Это самый опасный блок для масштаба. Если не исправить — при нагрузке деньги будут теряться, дублироваться или рассогласовываться.
2.1. updateBalance обходится в 14+ местах (H1 из вашего AUDIT)
exchange.ts (7 мест), gameTasks.ts, tasks.ts, tournaments.ts, wars.ts, cleanup.ts, crons.ts — все используют прямой prisma.user.update({ balance: { increment } }).
Последствия: Не отслеживается эмиссия, не обновляется лига, не считается totalEarned/totalSpent, не инвалидируется кеш.
Решение: Это ваш U6 в RECONSTRUCTION_TASKS — согласен, что это фундамент. Но я рекомендую поднять приоритет выше U5 (войны UI).
2.2. purchaseAttempts — не атомарная операция
Файл: backend/src/services/attempts.ts, строки ~46–50
updateBalance() и prisma.user.update({ attempts }) — две отдельные операции. При сбое между ними: деньги списаны, попытки не добавлены.
Решение: Обернуть в единую prisma.$transaction.
2.3. joinBattleSession — многошаговая операция без транзакции
Файл: backend/src/services/game/session.ts
Последовательность: проверка → useAttempt → updateBalance → session.update — каждый шаг отдельно. Два одновременных join могут создать гонку.
2.4. finishSession — не атомарная
Файл: backend/src/services/game/finish.ts
processPayouts → обновление сторон → обновление сессии → ELO → достижения — цепочка без единой транзакции. Сбой в середине = рассогласованное состояние (деньги начислены, но сессия не закрыта, или наоборот).
Решение для 2.2–2.4: Использовать prisma.$transaction с явным isolation level (Serializable для финансовых операций). Добавить version-field или optimistic locking для критичных записей.
2.5. Гонка при ходах (WebSocket)
game:move читает сессию из БД → валидирует ход → обновляет FEN/PGN. Нет блокировки. Два хода от двух вкладок одного игрока (или retry) могут интерливить.
Решение: Redis distributed lock на session:lock:${sessionId} перед обработкой хода (NX, TTL 5 сек).

БЛОК 4: ПОДГОТОВКА К МАСШТАБУ НА 1 МИЛЛИОН (закладывать сейчас)
Ваш PERFORMANCE.md уже описывает этапы масштабирования — план хороший. Но есть вещи, которые нужно архитектурно заложить сейчас, чтобы потом не переписывать.
4.1. Distributed lock для finishSession (multi-instance)
Сейчас timer expiry через Redis keyspace notification приходит на все инстансы. Каждый пытается вызвать finishSession. Проверка status !== IN_PROGRESS в начале — не достаточно (race condition между проверкой и обновлением).
Решение: SET NX session:finish:${sessionId} 1 EX 10 перед finishSession. Только один инстанс выиграет lock.
4.2. spectatorRooms — утечка памяти
Map&lt;sessionId, Set&lt;socketId&gt;&gt; растёт бесконечно. Ключи сессий никогда не удаляются (удаляются только socket-id из sets). Очистка каждый час при &gt;500 — костыль, не решение.
Решение: Удалять ключ сессии при finishSession. Или использовать Redis вместо in-memory Map.
4.3. Буферизация ходов (Redis вместо PostgreSQL на каждый ход)
Каждый ход = 2-3 записи в PostgreSQL (FEN/PGN + timeLeft + cache). При 10,000 партий — это 20,000+ writes/мин.
Решение (закладывать сейчас):
Текущий FEN/PGN хранить в Redis (уже частично делается через cacheSession)
В PostgreSQL писать только при завершении партии (финальный PGN)
timeLeft — только Redis, не PostgreSQL на каждый ход
4.4. Battles:list — дифференциальные обновления
Сейчас при каждом создании/присоединении к батлу полный список шлётся всем в лобби. При 10,000 онлайн — это колоссальный трафик.
Решение: Отправлять battles:added / battles:removed вместо полного списка. Это описано в PERFORMANCE.md — нужно реализовать.
4.5. PgBouncer — добавить в docker-compose.yml сейчас
При 3+ инстансах бэкенда Prisma connection pool × N = сотни соединений к PostgreSQL. PgBouncer — обязательный слой.

БЛОК 6: РЕКОМЕНДАЦИИ ПО АРХИТЕКТУРЕ «НА ВЫРОСТ»
6.1. Event-driven вместо direct calls (закладывать сейчас)
Сейчас finishSession вызывает напрямую: updateBalance → applyReferralIncome → checkGameAchievements → checkGameTasks → ELO → tournaments. Это монолитная цепочка — при добавлении новой фичи (например, уведомления) нужно менять finish.ts.
Решение: Внутренний EventEmitter (или Redis pub/sub):
gameFinished → [
  payoutService.handle,
  referralService.handle,
  achievementService.handle,
  tournamentService.handle,
  notificationService.handle  // легко добавить
]
Это не микросервисы — это просто развязка внутри монолита. Позже легко вынести в отдельные сервисы.
6.2. Optimistic locking для финансовых операций
Добавить version поле в User и проверять при updateBalance:
UPDATE users SET balance = balance + X, version = version + 1
WHERE id = ? AND version = ?
Если version не совпал — retry. Это предотвращает race conditions без тяжёлых блокировок.
6.3. Idempotency keys для критических операций
Покупка попыток, join к батлу, execute ордера биржи — все должны быть идемпотентны. Клиент отправляет уникальный idempotencyKey, сервер проверяет через Redis (TTL 5 мин).
6.4. Structured logging с correlation ID
Сейчас логи через Winston без correlation. При нескольких инстансах невозможно отследить запрос от начала до конца.
Решение: Middleware, который генерирует requestId и пробрасывает через весь стек (включая WebSocket events).

БЛОК A: ИГРОВАЯ МЕХАНИКА — что сломано и что нужно
A1. Jarvis крашится при старте игры
Корень проблемы найден. Цепочка такая:
HomePage.tsx вычисляет jarvisCfg = JARVIS_LEVELS[jarvisLevel - 1] и открывает GameSetupModal
После выбора цвета/времени отправляется game:create:bot через WebSocket
Бэкенд создаёт сессию, возвращает её через callback
GamePage.tsx ищет сессию в store: sessions.find(s =&gt; s.id === sessionId) ?? sessions[0]
Проблема: Если новая сессия ещё не попала в store (race condition между callback и обновлением store), страница не находит сессию → session = undefined → navigate('/', { replace: true }) — выбрасывает на главную.

Решение:
Добавить задержку/retry для поиска сессии в store
Добавить guard на JARVIS_LEVELS index bounds
Добавить React Error Boundary на GamePage

Проблемы:
Уровни 16–20 одинаковые по силе. UCI_LimitStrength = false, Skill Level = 20 — это означает, что Stockfish играет на полную силу на всех пяти уровнях. Разница только в movetime (4–6 секунд), но Stockfish на полной силе находит лучший ход за 1–2 секунды — дополнительное время практически ничего не меняет.
Уровни 1–10 слишком слабые. Elo 1320 на уровне 1 — это уровень абсолютного новичка. Но Elo 2200 на уровне 10 — это уже сильный любитель. Прыжок от 10 к 15 (2200→2700) — огромный, а от 15 к 20 — плоский.
Elo выше 2850 для Stockfish бессмысленны — при UCI_LimitStrength = false Stockfish играет на ~3500+ Elo. Установка UCI_Elo = 3190 не применяется, потому что UCI_LimitStrength = false.
Random fallback: Если Stockfish pool перегружен или воркер таймаутит, возвращается случайный ход вместо ошибки. На высоких уровнях это создаёт впечатление, что Jarvis "тупит".
Рекомендуемое решение:
Уровни 1-5:   UCI_LimitStrength=true, Elo 800→1600, Skill 0-6
               → Новичок, делает грубые ошибки
               
Уровни 6-10:  UCI_LimitStrength=true, Elo 1700→2200, Skill 7-14
               → Средний игрок, тактика есть
Уровни 11-15: UCI_LimitStrength=true, Elo 2300→2700, Skill 15-19
               → Сильный клубный, потом мастерский уровень
Уровни 16-17: UCI_LimitStrength=true, Elo 2800→2900, Skill 20
               → Гроссмейстер
Уровни 18-19: UCI_LimitStrength=false, depth 18-22, movetime 5-10 сек
               → Суперкомпьютер, но ограничен глубиной
Уровень 20 (Mystic): UCI_LimitStrength=false, depth 30+, movetime 30 сек
               → Практически непобедим. Чемпион мира проиграет.
Ключевое: для "легендарных" уровней нужно увеличить depth и movetime до 15–30 секунд, а не 6. Stockfish на полной силе с depth 30 и movetime 30000 — это уровень, с которым Магнус Карлсен проиграет.
A3. История партий и просмотр прошлых игр — ПОЛНОСТЬЮ СЛОМАНА
Корень проблемы: Несовпадение формата данных между бэкендом и фронтендом.
Бэкенд (GET /profile/games) возвращает плоский объект:
{ "sessionId": "...", "result": "WON", "opponent": {...}, "pgn": "1.e4 e5...", "bet": "1000" }
Фронтенд (ProfilePage.tsx) ожидает вложенный объект:
{ "id": "...", "status": "WON", "session": { "id": "...", "pgn": "...", "sides": [...] } }
Результат:
g.status → undefined (бэкенд шлёт result, не status) → неправильные цвета win/draw/lose
g.session?.pgn → undefined (нет вложенного session) → кнопка "Replay" никогда не появляется
g.session?.sides → undefined → имя противника не отображается
g.id → undefined → нестабильные React-ключи, мерцание
ELO-график на вкладке "Info" использует g.status === 'WON' → график пустой
Решение: Привести в соответствие — либо бэкенд переделать на вложенную структуру, либо (проще) фронтенд адаптировать под плоский формат. Я рекомендую фронтенд, потому что плоский формат бэкенда — более эффективный.
A4. Badge-система Jarvis — работает, но есть нюансы
Хорошая новость: бэкенд-логика начисления badges корректна. В finish.ts при победе над ботом:
Badge добавляется в jarvisBadges[]
Дата записывается в jarvisBadgeDates
jarvisLevel увеличивается на 1
Но проблема: Поскольку игра с Jarvis крашится (A1), badge-система фактически не работает — игрок не может завершить партию и получить badge.
Также: На HomePage игрок всегда стартует на своём текущем уровне (jarvisLevel), а не на jarvisLevel + 1. Бэкенд разрешает jarvisLevel + 1, но фронтенд не использует эту возможность. То есть после прохождения уровня 5, игрок должен начинать уровень 6, но фронтенд ставит ему снова уровень 5.
Решение: HomePage.tsx должен отправлять botLevel: jarvisLevel + 1 (но не выше 20), или jarvisLevel если все пройдены.
A5. Награды за Jarvis — регрессия уровней 11–20
В config.ts и .env.example:
BOT_REWARD_10 = 50,000 ᚙ   (в config.ts, отличается от .env!)
BOT_REWARD_11 = 11,000 ᚙ   ← в 4.5 раза МЕНЬШЕ чем уровень 10
BOT_REWARD_12 = 12,000 ᚙ
...
BOT_REWARD_20 = 20,000 ᚙ   ← в 2.5 раза МЕНЬШЕ чем уровень 10
Это точно баг — уровни 11–20 должны давать больше, а не меньше. Предлагаемая шкала:
1: 1K → 5: 5K → 10: 15K → 15: 50K → 18: 200K → 19: 500K → 20 (Mystic): 1,000K
A6. Рематч — баг с таймерами
В GamePage.tsx при ремотче timeSeconds берётся из оставшегося времени (botSide?.timeLeft). Если часы сели до 30 секунд, бэкенд сбрасывает до 600 (по ограничению &lt; 60). Игрок ожидает 3-минутную партию, получает 10-минутную.
БЛОК B: ДИЗАЙН И UX — проблемы
B1. Просмотр чужого профиля — не работает
/profile/:userId всегда показывает свой профиль. Нет запроса profileApi.getUser(viewedUserId) для других игроков. Бэкенд GET /profile/:userId возвращает ограниченные данные (stats), но фронтенд их не использует.
B2. VictoryScreen — подключена но не рендерится
В GamePage.tsx импортирован VictoryScreen, установлен showVictory, но нет &lt;VictoryScreen /&gt; в JSX. V3-анимации победы не работают.
B3. Скины бота не сохраняются
createBotSession не читает boardSkinUrl/pieceSkinUrl пользователя. Скины видны только в BATTLE-сессиях, не в BOT-играх. (M5 из AUDIT)
